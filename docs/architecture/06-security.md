# 06 — Security & Trust

> CTO 책임 영역(CTO_CHARTER: Security). AI 직원에게 회사의 기억과 권한을 위임하므로
> 신뢰(Trust)는 제품의 일부다.

## 위협 모델 (핵심)

| 위협 | 영향 | 완화 |
|---|---|---|
| 테넌트 간 데이터 유출 | 치명 | `brand_id/customer_id` 격리, Row-Level Security 후보 |
| BYOK 키 유출 | 치명 | 키는 전용 시크릿 저장소(KMS/Vault), DB에 평문 저장 금지 |
| Skill의 과도 권한/부작용 | 높음 | Sandbox 격리(Skill OS 3단계), 최소권한 가드레일 |
| 프롬프트 인젝션 → 메모리 오염 | 높음 | 쓰기 권한 분리, Memory write는 인증 직원만 + 리뷰 |
| 예산 우회 → 원가 폭증 | 높음 | 모든 호출 Model Gateway 강제 통과, 예산 사전 체크 |
| 감사 부재 | 중간 | 모든 상태 전이/권한 행위 `AuditEvent` 기록 |

## 원칙

1. **최소 권한.** 직원은 `memory_scope`가 허용한 Brand Memory만 접근. Skill은 매니페스트가 선언한 권한만.
2. **시크릿 분리.** BYOK 키·자격증명은 코드/DB/로그에 절대 남기지 않음. `.env`는 커밋 금지([.gitignore](../../.gitignore)).
3. **쓰기 게이트.** Brand Memory 쓰기는 읽기보다 엄격. 인증 + 감사 필수.
4. **격리.** Skill 실행은 Sandbox 우선. 부작용·네트워크·파일 접근은 명시 허용만.
5. **추적 가능성.** 누가/무엇을/언제 — 모든 민감 행위는 감사 로그.

## 멀티테넌시
- 데이터 격리를 1일차 설계에 포함(스키마의 테넌트 키 + 정책). 사후 추가는 위험.
- 최종 저장소/RLS 방식은 [ADR 0003](../adr/0003-tech-stack.md)에서 확정.

## 이번 Sprint 경계
- 인증/로그인은 **구현하지 않음**(범위 밖). 단, 데이터 모델·권한 경계는 *지금* 설계에 반영.
- 시크릿 관리 방식은 ADR 후보로 기록, 구현은 Sprint 1+.
